# Step 05：确定性随机数、ID 与运行上下文

状态：Completed  
日期：2026-06-12

## 范围与退出门槛

本步骤建立所有未来规则随机性的唯一入口、运行生成 ID 和可恢复 RunContext，不把它接入尚未实现的 Environment。

退出门槛：

- 同 seed 随机序列一致，不同 seed 有差异。
- 运行生成 ID 唯一且可恢复。
- manifest 保存重放所需元数据。
- RunContext 可以序列化和恢复。

## 改动

- 新增 `core/random.py`：项目自有 PCG32 确定性随机数实现。
- 扩展 `core/ids.py`：添加 Action、Event、Experience 顺序 ID 生成器及快照恢复。
- 新增 `core/run_context.py`：
  - 配置规范化与 SHA-256 摘要。
  - 默认稳定 Run ID。
  - seed、schema version、配置摘要、随机状态和 ID 状态。
  - JSON 兼容 manifest 与恢复。
- 新增 `tests/test_run_context.py`。

## 代码解释

### `DeterministicRandom`

项目没有直接使用 Python 全局 `random`。`DeterministicRandom` 提供：

- `random`
- `randint`
- `choice`
- `weighted_choice`
- JSON 兼容的 snapshot 与恢复

使用项目自有 PCG32 状态转换，避免运行规则散落未受控随机源，也避免 checkpoint 依赖 pickle。

### `DeterministicIdGenerator`

按类型保存独立计数器，生成：

- `action_000001`
- `event_000001`
- `experience_000001`

计数器进入 run manifest，恢复后继续生成而不重复。

### `RunContext`

RunContext 聚合一次运行的确定性服务。身份字段使用冻结 dataclass，不能在运行中替换；其中包含的随机数和 ID 生成器按确定性规则推进。默认 Run ID 由 seed 和配置摘要生成；负数 seed 使用 `n` 前缀，仍满足稳定命名约定。

配置摘要通过规范化 JSON 和 SHA-256 生成，字典键顺序不会影响摘要。

## 关键分析与取舍

- 没有使用 `random.Random`，因为自有小型 PRNG 更容易形成跨 checkpoint 的稳定、JSON 兼容状态。
- RunContext 身份被冻结，但内部确定性服务可推进；后续由 Environment 生命周期持有。
- 同配置与 seed 默认得到同 Run ID，便于重放识别；输出目录防覆盖策略留给 Storage 步骤。
